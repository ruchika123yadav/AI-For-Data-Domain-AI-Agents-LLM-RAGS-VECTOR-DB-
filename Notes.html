01

Artificial Intelligence: The Basics

AI is a big umbrella — and LLMs are just one branch sitting inside it. Get this map right and everything else clicks.

Artificial Intelligence is the science of making machines do tasks that normally need human intelligence — understanding language, recognising images, making decisions, or learning from experience.

💡 Simple Example

Think of a child learning to recognise a dog. You show many photos, and slowly the child learns the pattern. AI works the same way — we show it lots of examples and it learns the pattern, instead of being told every rule.

Machines learn patterns from data instead of following fixed, hand-written rules.

02

The "Before" — What Was the Problem?

Before LLMs we used traditional NLP. Tools like older Siri or 2015 Google Translate got confused easily. Three big flaws explain why.

FLAW 01
"Goldfish" Memory

Models read text in tiny windows. In a long paragraph, by the end they'd forgotten the beginning.

FLAW 02
Word Confusion

Struggled with words whose meaning changes with context (polysemy).

FLAW 03
Hyper-Specialisation

One model per task — a separate one for sentiment, translation, summarising. No multitasking.

💡 Simple Example

"I need a bank loan" vs "I sat by the river bank." Old AI couldn't tell the two "banks" apart.

🎯 Interview Point

Remember the three flaws: (1) no long-term context, (2) can't read meaning by context, (3) one model per task. They answer "Why did we need LLMs?"

03

What is an LLM?

An LLM (Large Language Model) is an AI trained on huge amounts of text to understand, process and generate human-like language.

Why "Large"?

Large dataset — it learns from billions of pages of text. Large parameters — its internal "settings" for making decisions, often hundreds of billions.

⭐ Key Idea

At its core, an LLM is a very advanced autocomplete engine. It doesn't "think" or "feel" — it calculates the mathematical probability of the next word based on everything it has read.

05

How LLMs Replaced Old, Task-Specific Models

Earlier, teams built small models from scratch for every task — slow, costly, hard to scale. LLMs changed that.

The approaches that came before

Bigrams / N-grams predicted the next word using only the last 1–2 words — tiny memory, no real understanding. RNNs read text one word at a time passing a "memory" along — better, but still forgot long-range info (the goldfish problem) and were slow to train.

ModelMemory / ContextMain weakness
Bigram / N-gramOnly last 1–2 wordsNo real context; breaks on long text.
RNNShort, fades over timeForgets long-range info; slow to train.
LLM (Transformer)Whole text at onceHeavy to train, but very powerful.
🎯 Interview Point

"Why did Transformers replace RNNs?" → RNNs read word-by-word and forgot long context, and couldn't be parallelised (slow). Transformers read the whole sentence at once using attention — capturing long-range context AND training far faster.

06

How LLMs Work: The Transformer

In 2017, Google's paper "Attention Is All You Need" introduced the Transformer — the backbone of every modern LLM (GPT, Claude, Gemini, Llama).

⭐ Key Idea

We keep the Transformer as the backbone of the LLM. It's what lets the model understand intent and context, not just individual words.

The high-level flow

1 · TokenizationBreak text into small units (tokens) → numbers the model can use.
→
2 · Self-AttentionEvery word looks at every other word to capture context.
→
3 · Train + AlignPre-training on data, then fine-tuning with human feedback.

Step 1 — Tokenization

💡 Simple Example

"unbelievable" might split into tokens: [ un ] [ believ ] [ able ]. The model never sees letters the way we do — it sees token pieces turned into numbers.

Step 2 — Self-Attention

Instead of reading word-by-word, the Transformer looks at the whole sentence at once and works out how every word relates to every other word.

💡 Simple Example

"The animal didn't cross the street because it was too tired." → attention links "it" → animal.

Change the ending to "…too wide." → attention shifts "it" → street. The model figures the link out from context automatically.

🎯 Interview Point

"What is self-attention?" → It lets the model weigh how important every other word is to each word, capturing context across the whole sentence at once.

Step 3 — Pre-training & Fine-Tuning

Pre-training ("the firehose"): the model reads a huge chunk of the internet and learns grammar, facts and reasoning by guessing the next hidden word trillions of times. Fine-tuning / alignment ("the guardrails"): human trainers guide it — using RLHF (Reinforcement Learning from Human Feedback) — so it behaves like a helpful assistant instead of blindly repeating internet text.

💡 Simple Example

Pre-training = a student reading the whole library. Fine-tuning = the teacher coaching them to answer politely and safely.

🎯 Interview Point

Know the full form: RLHF = Reinforcement Learning from Human Feedback. Pairing to remember — pre-training learns language from raw data; fine-tuning/RLHF aligns behaviour with human feedback.

Step 4 — Predicting Tokens (how the answer is actually built)

Here's the part that confuses most people. You type a question and a whole paragraph appears instantly — so it feels like the model wrote it all at once. But under the hood, an LLM can only ever produce one token at a time. It just does it so fast that we can't tell.

You send "I want to eat" → GPT predicts "chocolates", then "and", then "toffees", then ".", then a special <end> token to stop.

How the loop works

After all the attention layers have understood your input, the model's final stage — the LLM head — takes that contextual understanding and predicts the single most likely next token. Then comes the clever bit:

  1. Take the input tokens, run them through the model.
  2. The LLM head predicts one new token (e.g. "chocolates").
  3. Append that token to the input — now the input is one token longer.
  4. Feed the whole thing back in and predict the next token ("and").
  5. Repeat… "toffees" → "." → until the model predicts the special <end> token, which means "I'm done."

Tap below to watch it happen step by step:

Click "Predict next token" to start.
🔄 Each new token is appended, then the whole sequence is fed back in — this looping is called  auto-regressive generation.
⭐ Key Idea — the "LLM head"

After the attention layers build the meaning, a final layer called the LLM head (also "language-modelling head") maps that meaning to a probability score for every word in the vocabulary, and the highest-scoring one is picked as the next token.

🎤
The rapper analogy

Think of a freestyle rapper. They don't plan the whole verse — they say one word, that word shapes the next, and the next, building the line in real time. An LLM does the same: each token it picks becomes part of the input that shapes the following token. Super fast, but always one word at a time.

🧠 Memory Hook

"Predict → Append → Repeat → <end>". The model is like someone typing with autocomplete on steroids: pick a word, glue it on, look again, pick the next — stopping only when it hits the <end> sign. It feels instant, but it's a tight loop running thousands of times.

🎯 Interview Point

Three terms to drop confidently: auto-regressive (generates one token at a time, feeding each output back as input), the LLM head (final layer that picks the next token from the whole vocabulary), and the <end> / EOS token (special "end-of-sequence" token that tells the model to stop). If asked "if it's one token at a time, why is it so fast?" → because each prediction is a single forward pass on highly optimised hardware (GPUs), done thousands of times per second.

07

The Full LLM Pipeline — Step by Step

Let's trace one sentence all the way through an LLM, using your video's example: "I want to eat". Five stages turn your words into the model's answer.

Whiteboard: words split into tokens, then each token plotted as a point on the "meaning map".

Step 1 — Tokenization

Computers can't read letters — only numbers. Tokenization chops raw text into small units called tokens, and each token gets a unique ID from the model's dictionary.

IID 40
wantID 215
toID 12
eatID 89
💡 Simple Example

Old systems used whole words. Modern LLMs use sub-words: "eating" can split into [ eat ] [ ing ]. This lets the model handle words it has never seen before by reusing familiar pieces.

Step 2 — Static Embeddings

An ID like 40 still doesn't tell the computer what "I" means. So each ID is converted into an embedding — a long list of numbers (a vector) that acts like coordinates on a giant map of meaning. Words with similar meaning sit close together.

⭐ Key Idea — the "Static" flaw

A static embedding (like Word2Vec) gives a word the exact same coordinates everywhere. So "bank" in "deposit money in the bank" and "the river bank" gets the same value — even though they mean different things. That's the problem the next step fixes.

Step 3 — Contextual Embeddings (BERT)

Google's BERT (Bidirectional Encoder Representations from Transformers) pioneered contextual embeddings: instead of looking at a word alone, it looks at the words around it before deciding the final coordinates.

💡 Simple Example

In "Cat sat on mat", to understand "sat" the model draws links from "Cat", "on" and "mat" — realising a cat is performing the action onto a mat.

Change it to "The software sat on the server" and the embedding for "sat" shifts to a totally different spot on the map. Same word, different meaning, because context changed.

🎯 Interview Point

Static vs Contextual is a favourite question. One-liner: "Static embeddings give one fixed vector per word; contextual embeddings (BERT) change a word's vector based on its surrounding words." Also know: BERT = Bidirectional Encoder Representations from Transformers.

The complete flow (all 5 stages)

1
Input & Tokenization

Your prompt is split into tokens, each mapped to an ID.

"I want to eat" → [40, 215, 12, 89]
2
Embeddings + Positional Encoding

Tokens become vectors. A positional encoding is added so the model knows word order — "cat chased rat" ≠ "rat chased cat".

3
Self-Attention (the core engine)

For every word the model asks: "which other words matter most to understand this one?" and computes attention weights — contextual embedding at massive scale.

4
The Multi-Layer Stack

Dozens of attention layers stacked (GPT-3 has 96). Lower layers learn grammar & word links; higher layers learn abstract logic, themes and intent.

5
Output Generation

The final vector is mapped back to the vocabulary. The model picks the most likely next token, prints it, then repeats the whole loop word by word until done. (See Transformers → Step 4: Predicting Tokens for the full loop with a live demo.)

🎯 Interview Point

Two terms interviewers love here: Positional Encoding (tells the model word order) and "96 layers" (LLMs stack many attention layers; depth = more abstract understanding). And remember generation is one token at a time, in a loop.

Build It Yourself — Python + Gemini API

Now the hands-on part: setting up a Python project to actually call an LLM, using the Google Gemini API. Here's every step from your video, explained.

First — what is uv?

uv is a super-fast, modern tool for managing Python projects — installing packages, creating virtual environments, and handling versions. Think of it as a faster all-in-one replacement for pip + venv. It's written in Rust, so it's much quicker than the old tools.

💡 Simple Example

Old way needed several tools: pip to install, venv to isolate, requirements.txt to track. uv does all three, faster, with one tool.

Step 1 — Install uv & start the project

Terminal
$pip install uv
↳ installs the uv tool itself
$uv init
↳ creates a fresh project (pyproject.toml, main.py, .python-version)

Step 2 — Create the virtual environment

A virtual environment is an isolated box for this project's packages, so they don't clash with other projects. uv sync reads your project file and builds that box (the .venv folder) with the right packages.

Terminal
$uv sync
↳ creates .venv and installs everything listed in pyproject.toml

Step 3 — pyproject.toml is the brain 🧠

The pyproject.toml file is the brain of your project. It records the project name, the Python version, and every dependency (library) you need. When anyone runs uv sync, this file tells uv exactly what to install — so your setup is reproducible on any machine.

Your project structure

📁AI_DATA/
📁.venvthe isolated environment (made by uv sync)
📁LLMs/
📓api_calles.ipynbthe notebook where we run code
⚙️.envstores your secret API key (never share!)
📄.python-versionpins the Python version
⚙️pyproject.tomlthe brain — project + dependencies
🔒uv.lockexact versions locked for reproducibility

Step 4 — Make the notebook & pick a kernel

Create a .ipynb notebook file (here api_calles.ipynb), then select the Python kernel from your .venv. The kernel is the engine that actually runs your code — choosing the venv kernel means it uses the packages you installed.

VS Code: the notebook with the kernel selected (top-right) and the Gemini SDK code in a cell.

Step 5 — The code, line by line

api_calles.ipynb
from google import genai
import os
from dotenv import load_dotenv

# 1. Load the .env file (reads your secret key)
load_dotenv()

# 2. Create the client using your API key
client = genai.Client(api_key=GEMINI_API_KEY)

# 3. Send a prompt to the model
interaction = client.interactions.create(
    model="gemini-2.5-flash",
    input="Explain how AI works in a few words"
)

# 4. Print the model's text answer
print(interaction.output_text)

In plain English: load your secret key from .env, build a client with that key, send it a prompt (the model + your question), and print whatever the model replies.

⭐ Key Idea — why use a .env file?

Your API key is like a password — anyone with it can spend your money. We keep it in a separate .env file (never uploaded to GitHub) and load it with load_dotenv(), so the secret never sits inside your shared code.

🎯 Interview / Practical Point

Be ready to explain: virtual environment (isolated per-project packages), environment variables / .env (keeping secrets out of code), and the basic API call flow (client → prompt → response). These show you can actually ship an LLM app, not just describe one.

What is RAG? (Retrieval-Augmented Generation)

An LLM only knows what it was trained on — it can't see your company's PDFs, your private notes, or anything newer than its cutoff. RAG fixes this by letting the model look things up first, then answer.

📖
The open-book exam analogy

A plain LLM is a student answering from memory — it might forget or make things up. RAG turns it into an open-book exam: before answering, the student flips to the right page, reads it, and then writes the answer. The "book" is your own data.

The name spells out the three steps in order: Retrieval (find relevant info) → Augmented (add it to the question) → Generation (LLM writes the answer).

🧠 Memory Hook

Remember "R-A-G = Read, Add, Generate." First the system Reads your documents, then Adds the useful bits to your question, then the LLM Generates the final answer. Say it three times — it sticks.

The whole thing is built from two pipelines you set up at different times:

PIPELINE 1 · done once, in advance
Data Ingestion

Read your documents, chop them up, turn them into vectors, and store them in a vector database. This builds your "knowledge base".

PIPELINE 2 · runs on every question
Retrieval + Generation

A user asks something → find the closest chunks from the knowledge base → feed them to the LLM as context → return the answer.

🎯 Interview Point

One-line definition to memorise: "RAG retrieves relevant chunks from an external knowledge base and adds them to the prompt, so the LLM answers from real data instead of only its training memory." Top benefit phrases: reduces hallucinations, uses private/up-to-date data, no expensive re-training.

12

Pipeline 1 — Data Ingestion

This is the "prep the open book" stage. You run it once (or whenever documents change) to build your searchable knowledge base.

Ingestion: PDF → extract → split into chunks → embedding API → vectors → stored in semantic index (vector DB).
1
Extract Data

Pull the raw text out of your sources (PDFs, Word docs, web pages).

2
Split into Chunks

Break the long text into smaller pieces ("chunks"). Each chunk is a bite-sized passage the model can handle and search over.

e.g. one chunk ≈ a paragraph or a few sentences
3
Create Embeddings

Send each chunk through an embedding API → it becomes a vector (the list of numbers / coordinates of meaning you learned in Section 07).

4
Build the Semantic Index & Store

Save all the vectors into a vector database. This indexed store is your Knowledge Base.

stored: 1) the embeddings (vectors) 2) the original doc chunks
🧠 Memory Hook

Ingestion = "Extract → Split → Embed → Store". Picture a librarian who takes a big book, tears it into index cards, writes a secret "meaning code" on each card, and files them in a cabinet. That cabinet is your vector database.

⭐ Key Idea — why store the chunk too?

The vector database keeps both the vector AND the original text chunk. The vector is used to search; the text chunk is what actually gets fed to the LLM later. You search with math, but you answer with words.

13

Pipeline 2 — Retrieval, Augmentation & Generation

This runs live, every time a user asks a question. It finds the right chunks and hands them to the LLM.

Full picture: top row = ingestion (built earlier); bottom row = a user query flowing through search to the LLM and back as an answer.
1
User Query → Embedding

The user's question goes through the same embedding API used during ingestion, so it becomes a vector in the same "meaning space".

Important: query and chunks must use the SAME embedding model
2
Semantic Search (Retrieval)

The query vector is compared against all stored vectors. The database returns the chunks whose meaning is closest to the question.

3
Ranked Results — "top K"

Results come back ranked by similarity. You choose how many to keep — the top K (e.g. top 3 or top 5 most similar chunks).

4
Augmentation — build the prompt

Those top chunks become the context. They're combined with the user's question into one prompt for the LLM.

5
Generation — the answer

The LLM reads the prompt (question + retrieved context) and writes a grounded answer, which goes back to the user.

The magic formula — what actually goes into the LLM

Querythe user's question
+
Contextretrieved top-K chunks
+
Instructionse.g. "answer only from context"

So: Prompt = Query + Context + Instructions. This combined prompt is the heart of RAG — it's what turns a generic LLM into one that answers from your data.

🧠 Memory Hook

"Q + C + I" → think "Quick Coffee, Instantly." Query, Context, Instructions — the three things every RAG prompt always contains. The instruction part is what keeps the model honest ("only use the context above").

🎯 Interview Point

Two terms they'll probe: "top K" (how many retrieved chunks you pass as context — more isn't always better, it costs tokens) and "grounding" (forcing the model to answer from the provided context to reduce hallucination). Be ready to write Prompt = Query + Context + Instruction on a whiteboard.

The 3 Big Knobs of a RAG System

When you actually build RAG, three choices matter most. Get these right and your system is good; get them wrong and answers turn vague or wrong.

🧠 Memory Hook

Remember the "C-E-V" trio (say it like "see-v"): Chunking, Embedding model, Vector database. Those are the three dials you tune in every RAG project.

Knob 1 — Chunking Strategy

How you cut documents decides what the search can find. Cut badly and a chunk loses its meaning. Main strategies, tap to compare:

Fixed-size chunkingsimple but flawed
Cut every N characters/words. Easy, but it can slice a sentence or idea in half, breaking the meaning at the boundary.
Hierarchical chunkingkeeps structure
Splits along the document's natural structure — sections, sub-sections, paragraphs — so chunks respect headings and hierarchy.
Semantic chunkingmeaning-aware
Splits where the topic changes, so each chunk holds one coherent idea. Smartest, but more work to set up.

Popular frameworks that handle chunking for you: LangChain, LlamaIndex, and others.

Knob 2 — Embedding Model

This decides how well "meaning" is captured as vectors. A better embedding model = better search matches. Common choices:

  • OpenAI text-embedding models
  • Gemini embeddings (Google)
  • Sentence Transformers (open-source, runs locally / free)

Knob 3 — Vector Database

Where vectors live and get searched fast. Choice depends on scale, cost, and whether you want self-hosted or cloud:

  • ChromaDB — lightweight, great for learning & small projects
  • FAISS — fast, from Meta, runs locally
  • Pinecone — managed cloud service
  • Elastic, Azure AI Search — enterprise options
🎯 Interview Point

If asked "what affects RAG quality the most?" → lead with chunking strategy (most underrated), then embedding model and vector DB. Mention the fixed-size drawback (cuts ideas mid-way) — it shows real hands-on understanding, not just buzzwords.

16

Vector Database — The Heart of RAG

You met vector DBs as one of the "3 knobs". But this is where most interview questions and real confusion live — especially the big one: "aren't we re-embedding everything on every query?" Let's clear it all up.

What's actually stored inside

Each row = a Context (the embedding vector, e.g. [0.1, -0.99, …1]) paired with its Content (the original chunk / source, e.g. a PDF page). The vector is for searching; the content is what gets sent to the LLM.

Think of it as a two-column table: one column holds the vector (math you search with), the other holds the original text (words you answer with). They're linked, so once search finds the right vector, it can instantly fetch the matching text.

How semantic search actually finds the answer

The query "How much sales we made in 2020" becomes a vector, then the DB finds the nearest stored vectors in meaning-space and returns the Top 2 rows.
💡 Simple Example — the "closeness" idea

Every chunk is a point (or arrow) in a huge space. Chunks about similar topics sit close together. Your question is also turned into a point — the DB just grabs the few points nearest to it. "Nearest" = most similar meaning. That's the whole trick.

⚡ Your big question: are we re-embedding everything every time?

Short answer: No. This is the single most important efficiency idea in RAG, so let's walk it slowly.

1
The PDF is embedded ONCE (during ingestion)

When you first load the PDF, you split it into chunks and embed each chunk one time. Those vectors are saved in the vector DB forever (until docs change). You never re-embed the PDF again.

2
On each query, you embed ONLY the question

When a user asks something, you embed just that one short question (a handful of tokens) — not the whole PDF. One tiny embedding call.

3
Search compares vectors — no LLM, no tokens

Finding the nearest chunks is pure math (comparing numbers). It does not call the LLM and costs zero LLM tokens.

4
Only the Top-K chunks go to the LLM

The DB returns, say, the top 3 matching chunks. Only those few chunks become context for the LLM — not the entire document.

So to directly answer your two doubts:

  • "Do we re-create the embeddings each query?" → No. Chunk embeddings are made once and reused. Only the small query is embedded fresh each time.
  • "Does it reduce the tokens sent to the LLM?" → Yes, massively. Instead of stuffing the whole PDF into the prompt, you send only the few most relevant chunks.

Token cost: dump-the-whole-PDF vs RAG

Without Vector DB
~50,000
tokens sent to LLM every single query (entire PDF crammed in)
With Vector DB + RAG
~800
tokens sent (just the top-K relevant chunks)

(Numbers are illustrative.) Fewer tokens means cheaper, faster, and more accurate answers — the model isn't distracted by 49,000 irrelevant tokens. The vector DB is what makes this filtering possible.

Why a special vector DB? Why not a normal SQL database?

You could store vectors in any database — but normal databases are built to match exact values ("find rows where name = 'Aman'"). RAG needs to find things by meaning / closeness, which is a totally different job.

🗄️ Normal SQL DB (MySQL, etc.)
  • Finds exact matches ("=", "LIKE")
  • No idea what "similar meaning" is
  • To find nearest vector it must scan & compare every row — painfully slow at scale
  • Great for orders, users, prices
🧭 Vector DB (Pinecone, FAISS…)
  • Built for nearest-neighbour search by similarity
  • Special indexes (e.g. HNSW/ANN) find close vectors in milliseconds, even over millions
  • Returns ranked Top-K by closeness
  • Great for embeddings & semantic search
⭐ Key Idea — the optimisation

A vector DB is "optimised for connecting to the LLM" because it does the filtering before the LLM ever runs. It cheaply narrows millions of chunks down to the 3–5 that matter, so the expensive LLM only ever reads a tiny, highly-relevant slice. That's the whole efficiency win.

🧠 Memory Hook

"Embed once, ask cheap." The PDF pays its embedding cost a single time at ingestion. Every query after that is cheap: tiny query embedding + free math search + a few chunks to the LLM. The vector DB is the librarian who already indexed every book, so each visitor only gets handed the 3 right pages.

🎯 Interview Point

Expect: "Why use a vector database instead of a regular one?" → because RAG needs similarity search (nearest-neighbour), not exact matching, and vector DBs have specialised ANN indexes for it. Bonus follow-up: "Do you re-embed documents per query?" → No — embeddings are computed once at ingestion and reused; only the query is embedded each time, and only top-K chunks reach the LLM. Nailing this shows you understand RAG's cost model.

LangChain — The Glue Framework

Building RAG by hand means wiring up loaders, splitters, embedding APIs, the vector DB, retrieval, and the LLM call yourself. LangChain is a framework that gives you ready-made building blocks for all of these, so you connect them like LEGO instead of writing everything from scratch.

🔌
The universal-adapter analogy

Every embedding provider, vector DB, and LLM has its own way of being called. LangChain is like a universal adapter / power strip: you plug your components into its standard sockets, and it handles the messy wiring so the pieces talk to each other smoothly.

What LangChain typically handles in a RAG app:

  • Document loaders — read PDFs, web pages, Notion, etc.
  • Text splitters — do the chunking strategies for you
  • Embeddings & vector stores — connect to OpenAI/Gemini and Chroma/FAISS/Pinecone with one interface
  • Retrievers — the "go fetch top-K chunks" step
  • Chains — link steps together (retrieve → build prompt → call LLM → return answer)
⭐ Key Idea — swap, don't rewrite

Because LangChain wraps everything behind a common interface, you can swap one piece (say, change vector DB from Chroma to Pinecone, or OpenAI embeddings to Gemini) by changing one line — the rest of your pipeline stays the same. (LlamaIndex is a popular alternative aimed especially at data/retrieval.)

🧠 Memory Hook

LangChain = "chain the links." A RAG flow is a chain of steps (load → split → embed → store → retrieve → prompt → generate). LangChain literally lets you chain these together with reusable links.
